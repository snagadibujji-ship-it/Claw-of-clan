## 1. Executive Overview

**GHIA Scout** is an advanced, autonomous AI-powered penetration testing CLI application. Built on top of the **Model Context Protocol (MCP)** toolchain and a custom agentic reasoning framework, it orchestrates complex security exploration workflows from natural language commands. 

Unlike traditional static security scanners that run rigid pre-defined checklists, GHIA Scout acts as a goal-oriented reasoning agent. It is designed to assist cybersecurity professionals, red teams, and security researchers by automating the entire assessment cycle—from reconnaissance to vulnerability verification, exploitation, and reporting—while maintaining strict evidence boundaries.

## 2. Architecture Overview

GHIA Scout's runtime is built around a goal-driven state search space, structured using an OODA (Observe-Orient-Decide-Act) blackboard model.

```
       [ USER GOAL ]
             │
             ▼
      ┌──────────────┐
      │  REASON      │ ◄───────┐
      │  (Blackboard)│         │
      └──────┬───────┘         │
             │                 │ Update Facts
             ▼                 │ & Intents
      ┌──────────────┐         │
      │  EXPLORE     │ ────────┘
      │  (Tool Execution)
      └──────────────┘
```

### 2.1 State Space & Blackboard Engine (`agent/solver.py`)
Rather than relying on fixed loops, GHIA Scout models penetration testing as a path search from **Origin** (entry point) to **Goal** (e.g., retrieving a flag, confirming a vulnerability).
*   **Facts**: Verifiable, objective observations verified by tool executions (e.g., "Port 80 is open", "Server header is Apache/2.4.41").
*   **Intents**: Intentional exploration pathways generated by the reasoner (e.g., "Scan port 8080", "Perform SQL injection testing on login parameter").
Each intent is executed exactly once and marked as `concluded` or `abandoned`, preventing infinite loops or repetitive redundant actions.

### 2.2 Evidence-Level Anti-Hallucination Gate
To counter LLM hallucinations (such as generating fake flags or false-positive vulnerability reports):
*   **The Conclusion Gate**: Any claim made by the explorer agent (e.g., a flag or data leak) is validated directly against raw tool output. If the string is not present verbatim, the finding is marked as `[Unverified]` and rejected.
*   **The Completion Gate**: The session cannot be marked complete unless the target goal is structurally verified by hard evidence.

### 2.3 Reflexion Engine (`agent/reflexion.py`)
When obstacles (e.g., WAF blocks, payload filters) are encountered, failures are classified into progressive levels (**L0 to L4**):
*   **L0 (Raw)**: Baseline payload delivery.
*   **L1 (Encoding)**: Standard URL, hex, or HTML entity obfuscation.
*   **L2 (Double-Write)**: Double-writing filters (e.g., `seselectlect`).
*   **L3 (Unicode/Alternative)**: Alternative syntax, comment injection, or character sets.
*   **L4 (Alternative Vector)**: Shifting focus to other input fields or endpoints.

## 3. Installation Guide

### Prerequisites
*   **Python**: Version 3.10 or higher.
*   **Node.js**: Version 20 or higher (required for external MCP servers like chrome-devtools).
*   **System Tools (Optional)**: `nmap` (highly recommended for network discovery).

### 3.1 Installation from PyPI
```bash
pip install ghia-scout
```

### 3.2 Installation from Source
Clone the repository and install in editable mode:
```bash
git clone https://github.com/snagadibujji-ship-it/Claw-of-clan.git
cd Claw-of-clan
pip install -e ".[dev,web,kb]"
```

## 4. Quick Start

Follow these four steps to start your first security assessment:

### Step 1: Configure the LLM Provider
Set your preferred LLM provider preset (which automatically configures base URLs and default models):
```bash
ghia-scout config provider minimax
# Or choose from: openai, deepseek, zhipu, moonshot, qwen, siliconflow
```

### Step 2: Set API Credentials
```bash
ghia-scout config set llm.api_key sk-your-api-key-here
```

### Step 3: Run the Environment Check
Verify your local environment dependencies and credentials:
```bash
ghia-scout doctor
```

### Step 4: Initiate an Assessment
Launch the interactive Terminal User Interface (TUI):
```bash
ghia-scout tui
```
Or start a target run directly from the CLI:
```bash
ghia-scout scan http://testphp.vulnweb.com
```

## 5. Configuration Guide

Settings are stored in `~/.ghia-scout/config.yaml` (Windows: `C:\Users\<Username>\.ghia-scout\config.yaml`).

### Core Parameters

| Key | Type | Default | Description |
|-----|------|---------|-------------|
| `llm.provider` | `str` | `openai` | Target LLM provider. |
| `llm.model` | `str` | `gpt-4o` | Model name. |
| `llm.api_key` | `str` | `""` | Authentication token for the provider. |
| `session.engine` | `str` | `solve` | Execution engine: `solve` (recommended) or `rounds`. |
| `session.max_rounds` | `int` | `20` | Max reasoning iterations per run. |
| `safety.enable_python_execute` | `bool` | `True` | Allow local Python payload generation/execution. |
| `safety.python_execute_restricted` | `bool` | `False` | Sandbox restrictions on file/network access. |

To update settings via CLI:
```bash
ghia-scout config set session.max_rounds 30
```

## 6. Plugin Guide

GHIA Scout features a low-overhead, modular plugin runtime (`plugins/`) to perform passive and active checks on target metadata.

### 6.1 Writing a Custom Plugin
Plugins inherit from the `BasePlugin` class and implement search/assertion hooks:
```python
from vulnclaw.plugins.base import BasePlugin, PluginContext, PluginReport

class HeaderSecurityPlugin(BasePlugin):
    def name(self) -> str:
        return "header-security"

    async def execute(self, ctx: PluginContext) -> PluginReport:
        report = PluginReport(plugin_name=self.name())
        # Check HTTP response headers
        headers = ctx.target_headers
        if "X-Frame-Options" not in headers:
            report.add_finding(
                title="Missing X-Frame-Options Header",
                level="L1",
                description="Clickjacking vulnerability check failed."
            )
        return report
```

### 6.2 Execution
Plugins execute automatically during the Orient stage, and all validated findings are consolidated into the session state.

## 7. MCP Guide

GHIA Scout exposes four Model Context Protocol (MCP) server configurations.

### 7.1 Built-in MCP Servers
*   **`fetch`**: Executes HTTP requests, parses responses, and supports cookies and authorization.
*   **`memory`**: Persists session facts, contexts, and timelines across multiple reasoning loops.

### 7.2 External MCP Servers
*   **`chrome-devtools`**: Launches a headless browser instance using Puppeteer to intercept dynamic frontend interactions and client-side scripts.
*   **`burp`**: Bridges GHIA Scout to a running Burp Suite proxy interface via SSE (Server-Sent Events) to capture and replay HTTP history.

Configure external servers in `config.yaml`:
```yaml
mcp:
  servers:
    chrome-devtools:
      enabled: true
      transport:
        type: stdio
        command: npx
        args: ["-y", "chrome-devtools-mcp@latest"]
```

## 8. Reporting Guide

At the end of every cycle, GHIA Scout automatically writes structured reports and reproducible scripts to the output directory:
1.  **Markdown Report (`report_<timestamp>.md`)**:
    *   **Executive Summary**: Target host, test tools, and results.
    *   **Vulnerability Inventory**: List of verified vulnerabilities with technical descriptions and proof of concept references.
    *   **Timeline**: Event audit trace showing precisely when each fact was confirmed.
2.  **PoC Directory (`pocs/poc_<vulnerability>.py`)**:
    *   Autonomous standalone Python scripts utilizing `urllib` or `requests` that duplicate the exact trigger conditions discovered by the agent.
